# 算法输入输出规范

适用范围：

- 算法同学
- 后端同学

目标：

- 明确后端如何调用算法
- 明确算法输入格式
- 明确算法输出格式
- 保证后续真实算法服务可替换当前 mock 算法网关

关联代码：

- `backend-service/src/main/java/com/coldchain/backend/service/algorithm/AlgorithmGateway.java`
- `backend-service/src/main/java/com/coldchain/backend/service/algorithm/AlgorithmEvaluation.java`
- `backend-service/src/main/java/com/coldchain/backend/service/algorithm/AlgorithmRecommendation.java`

## 1. 当前接入模式

当前后端使用的是：

- mock 算法网关

当前真实状态：

- 后端已经定义好了算法输入边界和输出结构
- 当前通过 mock 数据生成风险评估和路径建议
- 后续可以直接把 mock 网关替换成真实 Python HTTP 服务调用

## 2. 推荐真实接入方式

建议真实算法服务采用：

- 独立 Python 服务
- HTTP/JSON 交互
- 后端发请求，算法返回结果

不建议：

- 前端直接调算法
- 算法直接写业务数据库
- 在 Java 中直接嵌 Python 脚本执行

## 3. 算法输入规范

后端传给算法的推荐输入结构如下：

```json
{
  "vehicleCode": "CC-VA-01",
  "vehicle": {
    "vehicleCode": "CC-VA-01",
    "cargoType": "VACCINE",
    "cargoName": "疫苗",
    "safeTempMin": 2.0,
    "safeTempMax": 8.0,
    "status": "IN_TRANSIT"
  },
  "latestTelemetry": {
    "recordTime": "2026-05-18 09:25:00",
    "temperature": 7.5,
    "humidity": 70.0,
    "doorOpen": false,
    "speed": 35.0,
    "outsideTemp": 31.0,
    "lng": 116.397,
    "lat": 39.908,
    "remainingKm": 13.4,
    "trend": "逼近上限"
  },
  "telemetryHistory": [
    {
      "recordTime": "2026-05-18 09:00:00",
      "temperature": 4.6
    },
    {
      "recordTime": "2026-05-18 09:05:00",
      "temperature": 4.9
    }
  ],
  "alerts": [
    {
      "alertId": "ALT-20260518-001",
      "level": "HIGH",
      "alertType": "TREND_WARNING",
      "triggerTime": "2026-05-18 09:25:00"
    }
  ]
}
```

## 4. 输入字段说明

### 4.1 `vehicle`

用于提供静态业务上下文：

- 货物类型
- 安全温区
- 当前车辆状态

### 4.2 `latestTelemetry`

用于提供当前时刻最重要状态：

- 当前温度
- 当前湿度
- 当前车门状态
- 当前速度
- 当前外界温度
- 当前经纬度
- 当前剩余路程

### 4.3 `telemetryHistory`

用于提供时间序列：

- 温度连续变化
- 波动趋势
- 短时上升

当前建议：

- 先保证最近 30 分钟时间序列可用
- 时间粒度优先 5 分钟一条

### 4.4 `alerts`

用于提供当前已知异常上下文：

- 历史是否已有趋势预警
- 是否已出现高风险告警
- 是否已有车门事件

## 5. 算法输出规范

当前后端已经约定好的推荐输出结构如下：

```json
{
  "anomalyDetected": true,
  "anomalyType": "TREND_RISE",
  "anomalyReason": "连续升温且已触发多条预警，存在温控失稳风险",
  "predictedMinutesToLimit": 12,
  "riskScore": 86.5,
  "riskLevel": "HIGH",
  "riskLabel": "高风险",
  "algorithmVersion": "risk-v1",
  "algorithmSource": "PYTHON_HTTP",
  "recommendations": [
    {
      "planType": "REROUTE_COLD_STORAGE",
      "title": "改道最近冷库",
      "detail": "优先前往最近冷库进行临时控温，降低货损风险。",
      "estimatedCost": "增加 8 分钟路程成本",
      "estimatedBenefit": "预计 3 公里内恢复控温",
      "recommended": true
    },
    {
      "planType": "PRIORITY_DELIVERY",
      "title": "优先配送最近医院",
      "detail": "缩短高敏货物暴露时间，减少超温影响。",
      "estimatedCost": "需调整后续配送顺序",
      "estimatedBenefit": "可减少约 18 分钟暴露时间",
      "recommended": false
    }
  ]
}
```

## 6. 输出字段说明

### 6.1 异常判断部分

- `anomalyDetected`
  - 是否检测到异常

- `anomalyType`
  - 异常类型
  - 示例：`TREND_RISE`

- `anomalyReason`
  - 异常原因摘要

- `predictedMinutesToLimit`
  - 预计多长时间后超出安全阈值

### 6.2 风险评估部分

- `riskScore`
  - 风险分数

- `riskLevel`
  - 固定枚举：`LOW`、`MEDIUM`、`HIGH`

- `riskLabel`
  - 中文风险等级

### 6.3 算法追踪部分

- `algorithmVersion`
  - 模型版本号

- `algorithmSource`
  - 算法来源
  - 当前建议真实服务返回：`PYTHON_HTTP`

### 6.4 路径建议部分

`recommendations` 为数组，每个对象包含：

- `planType`
- `title`
- `detail`
- `estimatedCost`
- `estimatedBenefit`
- `recommended`

要求：

- 至少返回 1 条建议
- 最好返回 1 到 3 条候选方案
- 必须明确哪一条是推荐方案

## 7. 后端如何消费算法结果

后端收到算法输出后，主要做三件事：

1. 映射成风险评估接口返回
2. 映射成路径建议接口返回
3. 后续落库到：
   - `risk_assessments`
   - `route_plans`

算法同学不需要关心前端展示细节，但需要保证输出字段稳定。

## 8. 必须统一的规则

### 8.1 风险等级

只允许：

- `LOW`
- `MEDIUM`
- `HIGH`

### 8.2 时间格式

统一使用：

- `yyyy-MM-dd HH:mm:ss`

### 8.3 布尔值

统一使用：

- `true`
- `false`

不要返回：

- `0`
- `1`
- `"yes"`
- `"no"`

## 9. 算法失败时的约定

如果算法服务异常，后端仍需可运行。

因此建议算法接口的失败处理如下：

- 返回 HTTP 错误码
- 或返回统一失败结构

后端降级规则：

- 使用 mock 风险评估或规则评估
- 使用固定路径建议模板
- 不让前端直接崩溃

## 10. 当前结论

当前你们已经具备：

- 明确的后端算法调用边界
- 固定的输入结构
- 固定的输出结构
- 可替换 mock 网关的正式接口契约

算法同学现在可以直接按这份文档去实现真实 Python 服务，不需要等后端再改一次结构。
