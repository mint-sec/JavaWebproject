## 2. 项目定义

### 2.1 项目名称

冷链运输温控预警平台

### 2.2 一句话定义

对冷链运输过程中的温度、位置、车门状态等信息进行实时监控，在异常趋势出现时提前预警，并结合货物风险与路线约束给出调度建议。

### 2.3 MVP 主场景

为避免范围失控，MVP 只聚焦一个主场景：

- 主货物场景：疫苗冷链配送
- 安全温区：2°C - 8°C
- 车辆规模：5 辆模拟冷链车
- 调度范围：单车重规划优先，多车协同不做
- 数据来源：模拟数据，不接真实硬件

### 2.4 核心目标

- 展示车辆实时位置、温度、湿度、车门状态、剩余里程
- 对温度异常做阈值预警和趋势预警
- 输出风险评分和风险等级
- 在高风险情况下输出路径重规划建议
- 支撑前端大屏与历史告警页演示

## 3. 角色分工与职责边界

### 3.1 后端职责

- 搭建主业务服务
- 设计数据库与数据流
- 对外提供前端接口
- 接收模拟数据并落库
- 编排算法调用并保存结果
- 生成告警记录、风险记录、调度建议
- 提供历史查询与演示支撑接口

### 3.2 前端职责

- 监控大屏展示
- 温度曲线、车辆轨迹、告警面板展示
- 风险等级与调度建议展示
- 页面轮询或订阅后端接口

### 3.3 算法职责

- 生成模拟数据或提供模拟数据规则
- 实现异常检测逻辑
- 实现风险评分逻辑
- 实现单车路径重规划逻辑
- 输出结构化结果供后端接入

### 3.4 职责边界原则

- 前端不计算核心业务规则，只负责展示
- 算法不直接操作前端页面，也不直接写业务库
- 后端负责“接入、编排、存储、对外接口”这一层

## 4. 后端负责人开工前必须做的前置工作

### 4.1 技术方案定版

建议 MVP 采用：

- 主后端：Java Spring Boot
- 数据库：MySQL
- 缓存：Redis 可选，不作为首周阻塞项
- 算法服务：Python FastAPI 或 Flask，HTTP/JSON 调用
- 前后端联调：REST API
- 实时刷新：前端轮询优先，WebSocket 暂不强制

选择这套方案的原因：

- 你负责后端，仓库也以 JavaWeb 项目命名，Spring Boot 更贴合分工
- 算法同学大概率使用 Python，HTTP 调用是 3 周内最稳的集成方式
- 轮询实现比 WebSocket 更轻，足够支撑课程演示

### 4.2 工程骨架准备

你需要先完成：

- 初始化 Spring Boot 项目
- 规划分层结构：controller / service / repository / dto / entity / config
- 配置 `application-dev.yml`
- 接入 MySQL
- 统一接口返回结构
- 增加全局异常处理
- 增加日志规范和请求 trace id
- 写本地启动说明

### 4.3 数据库初版设计

首版至少落下面几类表：

- `vehicles`：车辆基础信息
- `cargo_profiles`：货物类型与安全温区配置
- `telemetry_records`：温湿度/GPS/车门/速度等时序数据
- `alerts`：告警记录
- `risk_assessments`：风险评估记录
- `route_plans`：当前路径与重规划建议

如时间足够，可补：

- `destinations`
- `simulation_tasks`
- `algorithm_call_logs`

### 4.4 接口规范先行

在写业务代码前，先输出一版接口文档：

- URL 命名
- 请求方式
- 入参和出参字段
- 时间格式
- 温度单位
- 风险等级枚举
- 告警等级枚举
- 错误码

这一步必须先做，因为它决定前端和算法如何同时并行开发。

### 4.5 本地联调环境

## 5. 你要如何对接前端

### 5.1 对接策略

当前前端原型已经固定了几个核心展示区：

- 车辆地图区域
- 温度折线图区
- 告警信息面板
- 顶部元信息与概览卡片

所以后端最稳的做法不是一开始拆太细，而是先提供一个“大屏聚合接口”，让前端能最快把假数据切成真数据。

### 5.2 推荐给前端的 MVP 接口

#### 1. 大屏聚合接口

- `GET /api/v1/dashboard/vehicles/{vehicleId}`

用途：

- 一次返回当前车辆大屏所需主要数据

建议返回结构：

```json
{
  "vehicleId": "CC-VA-01",
  "scene": "疫苗冷链配送",
  "systemTime": "2026-05-16 10:10:00",
  "safeRange": { "min": 2.0, "max": 8.0 },
  "summary": {
    "temperature": 7.5,
    "humidity": 70,
    "speed": 35,
    "doorOpen": false,
    "riskLevel": "HIGH",
    "riskLabel": "高风险",
    "remainingKm": 13.4,
    "eta": "2026-05-16 10:11:00",
    "trend": "逼近上限"
  },
  "route": {
    "status": "候选方案: 改道冷库",
    "currentPosition": { "lng": 116.397, "lat": 39.908 },
    "pathPoints": [],
    "destinations": []
  },
  "temperatureHistory": [
    { "time": "09:00", "temperature": 4.6 },
    { "time": "09:05", "temperature": 4.9 }
  ],
  "alerts": [
    {
      "level": "HIGH",
      "title": "高风险临界告警",
      "detail": "疫苗车厢温度接近安全上限，剩余路线较长。",
      "suggestion": "比较最近冷库改道方案与继续配送方案的综合成本。"
    }
  ]
}
```

#### 2. 历史温度轨迹接口

- `GET /api/v1/vehicles/{vehicleId}/telemetry/history?minutes=30`

#### 3. 告警列表接口

- `GET /api/v1/alerts?vehicleId=CC-VA-01&page=1&pageSize=20`

#### 4. 历史告警详情接口

- `GET /api/v1/alerts/{alertId}`

### 5.3 你和前端必须先说清楚的点

- 前端是否采用轮询，轮询间隔是 3 秒、5 秒还是 10 秒
- 页面展示用中文枚举还是英文枚举
- 地图坐标格式用 `lng/lat` 还是自定义点位
- 温度趋势图需要最近 30 分钟还是最近 1 小时
- 告警列表是否只显示最新 4 条
- 风险等级展示色：低/中/高 的颜色映射
- 大屏是否先只支持单车切换

## 6. 你要如何对接算法

### 6.1 推荐集成方式

MVP 推荐：

- 算法同学独立提供 Python HTTP 服务
- 后端通过 HTTP 调算法
- 后端负责把算法结果持久化并转给前端

不推荐 MVP 阶段做的方式：

- 直接在 Java 里嵌 Python
- 一开始就接 Kafka/Flink
- 让算法服务直接操作业务数据库

### 6.2 推荐算法输入契约

后端传给算法的最小输入建议为：

```json
{
  "vehicleId": "CC-VA-01",
  "cargoType": "VACCINE",
  "safeRange": { "min": 2.0, "max": 8.0 },
  "currentSnapshot": {
    "timestamp": "2026-05-16 10:10:00",
    "temperature": 7.5,
    "humidity": 70,
    "doorOpen": false,
    "speed": 35,
    "outsideTemp": 31,
    "remainingKm": 13.4
  },
  "recentTelemetry": [
    { "timestamp": "2026-05-16 09:45:00", "temperature": 6.1 },
    { "timestamp": "2026-05-16 09:50:00", "temperature": 6.8 },
    { "timestamp": "2026-05-16 09:55:00", "temperature": 7.2 }
  ],
  "routeContext": {
    "currentPoint": { "lng": 116.397, "lat": 39.908 },
    "destinations": [],
    "coldStorages": []
  }
}
```

### 6.3 推荐算法输出契约

算法至少返回：

```json
{
  "anomalyDetected": true,
  "anomalyType": "TREND_RISE",
  "anomalyReason": "连续升温且预计 12 分钟后越过上限",
  "predictedMinutesToLimit": 12,
  "riskScore": 86.5,
  "riskLevel": "HIGH",
  "riskLabel": "高风险",
  "recommendations": [
    {
      "planType": "PRIORITY_DELIVERY",
      "title": "优先配送最近医院",
      "estimatedImpact": "减少暴露时间 18 分钟"
    },
    {
      "planType": "REROUTE_COLD_STORAGE",
      "title": "改道最近冷库",
      "estimatedImpact": "3 公里内可恢复控温"
    }
  ]
}
```

### 6.4 你和算法必须先说清楚的点

- 算法服务是同步调用还是异步批处理
- 输入窗口长度是多少
- 时间粒度是 1 分钟、5 分钟还是 10 分钟
- 异常检测输出的是布尔值、分值还是分类标签
- 风险评分范围是 `0-100` 还是 `0-1`
- 风险等级如何从分值映射
- 路径优化输出一条建议还是多个候选方案
- 算法失败时后端如何降级

### 6.5 推荐降级策略

如果算法服务不可用，后端仍然要能跑演示：

- 规则阈值预警继续可用
- 风险评分可回退到简单规则模型
- 调度建议可返回固定模板建议
- 前端显示“算法结果暂不可用，当前为规则引擎结果”

## 7. 开发前必须三方拍板的事项

下面这些不先定，后面一定返工：

### 7.1 场景与范围

- 最终只做疫苗场景，还是要兼容生鲜/药品
- 只做单车演示，还是多车切换
- 是否做登录鉴权
- 是否做司机端/移动端

### 7.2 数据口径

- 温度单位固定为摄氏度
- 时间统一为 `yyyy-MM-dd HH:mm:ss`
- 经纬度精度保留几位
- 车门状态是布尔值还是枚举
- 剩余里程由谁计算：前端、后端还是算法

### 7.3 告警与风险规则

- 告警级别几档
- 风险级别几档
- 告警和风险是否一一对应
- 什么条件下创建一条新的告警
- 告警是否需要确认、消警、关闭状态

### 7.4 算法接入边界

- 后端是否持久化算法原始输出
- 后端是否有权对算法输出做二次加工
- 路径优化结果由谁负责解释文案
- 算法版本是否记录

### 7.5 联调机制

- 先 mock 接口还是先联真实算法
- 每天固定联调时间
- 使用什么接口文档工具
- 使用哪份演示数据作为联调基准数据

## 8. MVP 功能需求

### 8.1 监控大屏

必须支持：

- 展示当前车辆编号、系统时间、监控场景
- 展示当前温度、湿度、速度、风险等级、剩余里程
- 展示最近 30 分钟温度趋势
- 展示车辆当前位置和配送点
- 展示最新告警与处置建议

### 8.2 温控预警

必须支持：

- 阈值越界预警
- 连续升温趋势预警
- 告警分级展示
- 告警记录查询

### 8.3 风险评估

必须支持：

- 输出风险分数
- 输出风险等级
- 输出风险原因摘要

### 8.4 路径建议

MVP 只需支持：

- 当前路线继续执行
- 优先配送最近站点
- 改道最近冷库

无需支持：

- 多车联合调度
- 动态交通实时重规划
- 复杂运筹求全局最优

### 8.5 历史查询

必须支持：

- 查询历史温度记录
- 查询历史告警记录
- 查询单次风险评估结果

## 9. 非功能要求

- 接口响应时间：普通查询接口 < 1 秒
- 告警可见延迟：演示环境下 < 5 秒
- 服务可本地一键启动
- 即使算法服务挂掉，大屏也不能整体崩溃
- 所有时间、温度、风险字段命名统一

## 13. MVP 验收标准

项目可视为达到 MVP 验收，当且仅当：

- 前端大屏不再依赖硬编码假数据
- 后端可返回真实接口数据
- 算法可返回异常检测和风险评估结果
- 出现异常时页面能看到告警、风险等级和处置建议
- 至少完成 1 个完整演示链路：
  - 正常配送
  - 温度连续升高
  - 触发预警
  - 风险升高
  - 输出路径建议

## 14. 对后端负责人的最终行动建议

你现在最应该先做的不是直接写业务细节，而是按下面顺序推进：

1. 定技术栈：Spring Boot + MySQL + Python 算法 HTTP 服务
2. 定接口：先写大屏聚合接口和告警接口文档
3. 定表结构：先把 `vehicles`、`telemetry_records`、`alerts`、`risk_assessments` 落下来
4. 定三方口径：场景、字段、枚举、时间格式、轮询频率
5. 先用 mock 算法结果把前后端联通
6. 再替换成真实算法服务

如果必须压缩范围，优先保：

- 单一疫苗场景
- 单车演示链路
- 规则预警 + 简单风险评分 + 简单路径建议

优先砍掉：

- 权限系统
- 多车联合调度
- 真实硬件接入
- 复杂消息队列
- 复杂深度学习模型
