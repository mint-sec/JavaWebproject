# 新增功能前后端接口契约

适用日期：2026-06-03  
适用范围：登录注册、管理员后台管理（用户管理、车辆管理、告警处理）  
说明：本文以当前前端代码实现为准，后端尚未开发的部分请优先按本文接口契约落地。

## 1. 目标

本次新增前端功能主要包括：

- 用户登录
- 用户注册
- 管理员后台概览
- 用户管理
- 车辆管理
- 告警处理

前端已经完成页面和交互，当前使用本地 `localStorage` 模拟数据。后端接入时，请尽量保持以下接口结构不变，以减少前端改动成本。

## 2. 统一约定

### 2.1 接口前缀

- 建议统一前缀：`/api/v1`

### 2.2 鉴权方式

- 登录、注册接口不需要鉴权
- 进入后台后的接口统一使用：

```http
Authorization: Bearer {token}
```

### 2.3 通用返回格式

建议沿用现有项目文档中的统一格式：

```json
{
  "success": true,
  "message": "ok",
  "data": {}
}
```

失败时建议：

```json
{
  "success": false,
  "message": "用户名或密码错误",
  "data": null
}
```

### 2.4 时间格式

- 统一使用：`yyyy-MM-dd HH:mm:ss`
- 示例：`2026-06-03 09:25:00`

## 5. 认证接口

### 5.1 用户登录

- Method：`POST`
- Path：`/api/v1/auth/login`
- 用途：登录页提交用户名和密码

请求体：

```json
{
  "account": "admin",
  "password": "Admin123!"
}
```

字段说明：

- `account`：前端当前使用“用户名”登录，后端一期先按用户名处理即可
- `password`：密码明文传输，实际部署请配合 HTTPS

成功响应：

```json
{
  "success": true,
  "message": "ok",
  "data": {
    "userId": "USR-ADMIN-001",
    "username": "admin",
    "displayName": "admin",
    "role": "ADMIN",
    "roleLabel": "管理员",
    "token": "jwt-or-session-token",
    "loggedInAt": "2026-06-03 10:00:00"
  }
}
```

失败场景建议：

- 用户名不存在
- 密码错误
- 用户状态为 `已封禁`

### 5.2 用户注册

- Method：`POST`
- Path：`/api/v1/auth/register`
- 用途：注册页提交注册信息

请求体：

```json
{
  "username": "zhangsan",
  "phone": "13800000003",
  "email": "zhangsan@example.com",
  "password": "123456",
  "confirmPassword": "123456"
}
```

字段要求：

- `username`：必填，唯一
- `phone`：必填，唯一，11 位手机号
- `email`：必填，唯一
- `password`：必填，长度不少于 6 位
- `confirmPassword`：建议后端也校验一次

成功响应：

说明：前端当前注册成功后会自动登录，因此返回结构请与登录接口保持一致。

```json
{
  "success": true,
  "message": "ok",
  "data": {
    "userId": "USR-202606030001",
    "username": "zhangsan",
    "displayName": "zhangsan",
    "role": "USER",
    "roleLabel": "普通用户",
    "token": "jwt-or-session-token",
    "loggedInAt": "2026-06-03 10:05:00"
  }
}
```

失败场景建议：

- 用户名重复
- 手机号重复
- 邮箱重复
- 两次密码不一致

### 5.3 获取当前登录用户

- Method：`GET`
- Path：`/api/v1/auth/me`
- 用途：页面刷新后恢复登录态；管理员修改自己的角色或状态后刷新当前会话信息

成功响应：

```json
{
  "success": true,
  "message": "ok",
  "data": {
    "userId": "USR-ADMIN-001",
    "username": "admin",
    "displayName": "admin",
    "role": "ADMIN",
    "roleLabel": "管理员",
    "token": "jwt-or-session-token",
    "loggedInAt": "2026-06-03 10:00:00"
  }
}
```

鉴权失败建议返回 `401`。

### 5.4 退出登录

- Method：`POST`
- Path：`/api/v1/auth/logout`
- 用途：清理服务端会话或令牌黑名单

请求体：

```json
{}
```

成功响应：

```json
{
  "success": true,
  "message": "ok",
  "data": null
}
```

说明：如果后端使用纯 JWT 且不做服务端注销，本接口也可以保留为空实现。

## 7. 用户管理接口

### 7.1 修改用户角色或状态

- Method：`PATCH`
- Path：`/api/v1/admin/users/{userId}`
- 用途：管理员修改用户角色、封禁/解封用户

请求体：

```json
{
  "role": "ADMIN",
  "status": "启用中"
}
```

字段说明：

- `role`：必填，`ADMIN` 或 `USER`
- `status`：必填，`启用中` 或 `已封禁`

成功响应：

```json
{
  "success": true,
  "message": "ok",
  "data": {
    "id": "USR-OPS-001",
    "username": "operator",
    "displayName": "operator",
    "phone": "13800000002",
    "email": "operator@coldchain.local",
    "role": "ADMIN",
    "roleLabel": "管理员",
    "status": "启用中",
    "origin": "系统账号"
  }
}
```

联调说明：

- 如果修改的是当前登录用户本人，前端后续会刷新登录态
- 如果修改后用户被封禁，后端后续接口应返回 `401` 或 `403`

## 9. 告警处理接口

### 9.1 更新告警处理结果

- Method：`PATCH`
- Path：`/api/v1/admin/alerts/{alertId}`
- 用途：修改告警处理人、优先级、状态、备注

请求体：

```json
{
  "owner": "调度负责人",
  "level": "HIGH",
  "status": "处理中",
  "note": "已联系司机，等待进一步确认"
}
```

字段说明：

- `owner`：必填，处理人
- `level`：必填，`HIGH` / `MEDIUM` / `LOW`
- `status`：必填，`待处理` / `处理中` / `已处理`
- `note`：可空字符串

成功响应：

```json
{
  "success": true,
  "message": "ok",
  "data": {
    "id": "ADM-ALT-01",
    "title": "高温风险待处理",
    "level": "HIGH",
    "levelLabel": "高",
    "detail": "车辆 CC-VA-01 的应急改道方案仍待管理员确认。",
    "vehicleId": "CC-VA-01",
    "owner": "调度负责人",
    "status": "已处理",
    "note": "已确认改道并通知司机执行",
    "handledAt": "2026-06-03 10:40:00"
  }
}
```

处理规则建议：

- 当 `status = 已处理` 时，后端补齐 `handledAt`
- 当 `status != 已处理` 时，`handledAt` 可为空字符串

## 11. 联调注意事项

- 前端当前强依赖 `roleLabel` 和 `levelLabel`，后端最好直接返回
- 如果后端不想返回 `roleLabel`、`levelLabel`，需要提前通知前端补映射层
- 前端当前使用 `vehicleId` 和 `id`，请不要自行替换成 `vehicleCode`、`alertId` 后又不通知前端
- 注册成功后，前端预期直接进入登录态，所以注册接口请返回 session 信息
- 后台页面当前优先使用聚合接口 `GET /api/v1/admin/console`

## 12. 推荐开发顺序

建议后端按下面顺序开发，前端最容易联调：

1. `POST /api/v1/auth/login`
2. `POST /api/v1/auth/register`
3. `GET /api/v1/auth/me`
4. `GET /api/v1/admin/console`
5. `PATCH /api/v1/admin/users/{userId}`
6. `POST /api/v1/admin/vehicles`
7. `PUT /api/v1/admin/vehicles/{vehicleId}`
8. `DELETE /api/v1/admin/vehicles/{vehicleId}`
9. `PATCH /api/v1/admin/alerts/{alertId}`

如果后端后续想拆分成更细的列表接口，也可以再补充，但以上这些已经足够支撑当前前端页面完整联调。
