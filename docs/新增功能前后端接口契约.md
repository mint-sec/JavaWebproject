# 新增功能前后端接口契约

适用日期：2026-06-03  
适用范围：登录注册、管理员后台第一套结构、业务侧“我的车辆 / 我的告警”  
说明：本文以当前前端代码实现为准，后端尚未开发的部分请优先按本文接口契约落地。

## 1. 目标

当前前端已经落地的新增功能包括：

- 用户登录
- 用户注册
- 管理员后台
- 平台概览
- 用户管理
- 日志中心
- 服务监控
- 我的车辆
- 我的告警

其中有一条很重要的业务边界：

- 管理员后台不再负责车辆管理和告警处理
- 车辆维护、业务告警处理属于普通业务用户自己的工作区

## 2. 统一约定

### 2.1 接口前缀

- 建议统一前缀：`/api/v1`

### 2.2 鉴权方式

- 登录、注册接口不需要鉴权
- 登录后接口统一使用：

```http
Authorization: Bearer {token}
```

### 2.3 通用返回格式

建议统一：

```json
{
  "success": true,
  "message": "ok",
  "data": {}
}
```

失败示例：

```json
{
  "success": false,
  "message": "用户名或密码错误",
  "data": null
}
```

### 2.4 时间格式

- 统一使用：`yyyy-MM-dd HH:mm:ss`
- 示例：`2026-06-03 09:25:00`

## 5. 认证接口

### 5.1 用户登录

- Method：`POST`
- Path：`/api/v1/auth/login`
- 用途：登录页提交用户名和密码

请求体：

```json
{
  "account": "admin",
  "password": "Admin123!"
}
```

字段说明：

- `account`：前端当前使用“用户名”登录，后端一期先按用户名处理即可
- `password`：密码明文传输，实际部署请配合 HTTPS

成功响应：

```json
{
  "success": true,
  "message": "ok",
  "data": {
    "userId": "USR-ADMIN-001",
    "username": "admin",
    "displayName": "admin",
    "role": "ADMIN",
    "roleLabel": "管理员",
    "token": "jwt-or-session-token",
    "loggedInAt": "2026-06-03 10:00:00"
  }
}
```

失败场景建议：

- 用户名不存在
- 密码错误
- 用户状态为 `已封禁`

### 5.2 用户注册

- Method：`POST`
- Path：`/api/v1/auth/register`
- 用途：注册页提交注册信息

请求体：

```json
{
  "username": "zhangsan",
  "phone": "13800000003",
  "email": "zhangsan@example.com",
  "password": "123456",
  "confirmPassword": "123456"
}
```

字段要求：

- `username`：必填，唯一
- `phone`：必填，唯一，11 位手机号
- `email`：必填，唯一
- `password`：必填，长度不少于 6 位
- `confirmPassword`：建议后端也校验一次

成功响应：

说明：前端当前注册成功后会自动登录，因此返回结构请与登录接口保持一致。

```json
{
  "success": true,
  "message": "ok",
  "data": {
    "userId": "USR-202606030001",
    "username": "zhangsan",
    "displayName": "zhangsan",
    "role": "USER",
    "roleLabel": "普通用户",
    "token": "jwt-or-session-token",
    "loggedInAt": "2026-06-03 10:05:00"
  }
}
```

失败场景建议：

- 用户名重复
- 手机号重复
- 邮箱重复
- 两次密码不一致

### 5.3 获取当前登录用户

- Method：`GET`
- Path：`/api/v1/auth/me`
- 用途：页面刷新后恢复登录态；管理员修改自己的角色或状态后刷新当前会话信息

成功响应：

```json
{
  "success": true,
  "message": "ok",
  "data": {
    "userId": "USR-ADMIN-001",
    "username": "admin",
    "displayName": "admin",
    "role": "ADMIN",
    "roleLabel": "管理员",
    "token": "jwt-or-session-token",
    "loggedInAt": "2026-06-03 10:00:00"
  }
}
```

鉴权失败建议返回 `401`。

### 5.4 退出登录

- Method：`POST`
- Path：`/api/v1/auth/logout`
- 用途：清理服务端会话或令牌黑名单

请求体：

```json
{}
```

成功响应：

```json
{
  "success": true,
  "message": "ok",
  "data": null
}
```

说明：如果后端使用纯 JWT 且不做服务端注销，本接口也可以保留为空实现。

### 6.2 修改用户角色或状态

- Method：`PATCH`
- Path：`/api/v1/admin/users/{userId}`
- 用途：管理员修改用户角色、封禁或解封用户

请求体：

```json
{
  "role": "ADMIN",
  "status": "启用中"
}
```

字段说明：

- `role`：必填，`ADMIN` 或 `USER`
- `status`：必填，`启用中` 或 `已封禁`

成功响应：

```json
{
  "success": true,
  "message": "ok",
  "data": {
    "id": "USR-OPS-001",
    "username": "operator",
    "displayName": "operator",
    "phone": "13800000002",
    "email": "operator@coldchain.local",
    "role": "ADMIN",
    "roleLabel": "管理员",
    "status": "启用中",
    "origin": "系统账号"
  }
}
```

联调说明：

- 如果修改的是当前登录用户本人，前端后续会刷新登录态
- 如果修改后用户被封禁，后续接口建议返回 `401` 或 `403`
- 后端服务端建议同步记一条操作日志

- 用途：当前登录用户新增自己的车辆

请求体：

```json
{
  "vehicleId": "CC-VA-08",
  "cargoName": "疫苗",
  "status": "运输中",
  "driver": "李明",
  "route": "北京仓库 -> 区域医院"
}
```

成功响应：

```json
{
  "success": true,
  "message": "ok",
  "data": {
    "vehicleId": "CC-VA-08",
    "cargoName": "疫苗",
    "status": "运输中",
    "driver": "李明",
    "route": "北京仓库 -> 区域医院",
    "ownerUserId": "USR-OPS-001",
```

说明：

- `ownerUserId` 和 `ownerName` 建议由后端按当前登录用户自动补齐
- 如果 `vehicleId` 重复，建议返回失败

### 7.3 编辑我的车辆

- Method：`PUT`
- Path：`/api/v1/workspace/vehicles/{vehicleId}`
- 用途：当前登录用户编辑自己名下车辆

请求体：

```json
{
  "vehicleId": "CC-VA-08",
  "cargoName": "疫苗",
  "status": "待命中",
  "driver": "李明",
  "route": "北京仓库 -> 市医院"
}
```

成功响应：

```json
{
  "success": true,
  "message": "ok",
  "data": {
    "vehicleId": "CC-VA-08",
    "cargoName": "疫苗",
    "status": "待命中",
    "driver": "李明",
    "route": "北京仓库 -> 市医院",
    "ownerUserId": "USR-OPS-001",
    "ownerName": "operator",
    "updatedAt": "2026-06-03 10:30:00"
  }
}
```

说明：

- 路径参数 `{vehicleId}` 为原车辆编号
- 请求体中的 `vehicleId` 为修改后的车辆编号，允许修改，但必须唯一
- 如果操作的不是当前用户自己的车辆，建议返回 `403`

### 7.4 删除我的车辆

- Method：`DELETE`
- Path：`/api/v1/workspace/vehicles/{vehicleId}`
- 用途：当前登录用户删除自己名下车辆

成功响应：

```json
{
  "success": true,
  "message": "ok",
  "data": null
}
```

说明：

- 如果车辆不存在，建议返回 `404`
- 如果操作的不是当前用户自己的车辆，建议返回 `403`

### 7.5 更新我的告警处理结果

- Method：`PATCH`
- Path：`/api/v1/workspace/alerts/{alertId}`
- 用途：当前登录用户修改自己负责告警的处理人、优先级、状态、备注

请求体：

```json
{
  "owner": "admin",
  "level": "HIGH",
  "status": "处理中",
  "note": "已联系司机，等待进一步确认"
}
```

字段说明：

- `owner`：必填，处理人
- `level`：必填，`HIGH` / `MEDIUM` / `LOW`
- `status`：必填，`待处理` / `处理中` / `已处理`
- `note`：可空字符串

成功响应：

```json
{
  "success": true,
  "message": "ok",
  "data": {
    "id": "ADM-ALT-01",
    "title": "高温风险待处理",
    "level": "HIGH",
    "levelLabel": "高",
    "detail": "车辆 CC-VA-01 的应急改道方案仍待管理员确认。",
    "vehicleId": "CC-VA-01",
    "owner": "admin",
    "ownerUserId": "USR-ADMIN-001",
    "status": "已处理",
    "note": "已确认改道并通知司机执行",
    "handledAt": "2026-06-03 10:40:00",
    "domain": "BUSINESS"
  }
}
```

处理规则建议：

- 当 `status = 已处理` 时，后端补齐 `handledAt`
- 当 `status != 已处理` 时，`handledAt` 可为空字符串
- 如果告警不属于当前登录用户，建议返回 `403`
- 如果 `domain = PLATFORM`，不允许通过该接口处理

## 9. 联调注意事项

- 前端当前强依赖 `roleLabel`、`levelLabel`，后端最好直接返回
- `serviceMonitors[].tone` 不是展示文案，而是前端样式值，请按 `normal / warning / pending` 返回
- `users`、`vehicles`、`alerts` 中的字段名请不要随意替换，比如不要把 `vehicleId` 改成 `vehicleCode`、把 `id` 改成 `alertId` 但不通知前端
- 注册成功后，前端预期直接进入登录态，所以注册接口请返回 session 信息
- 管理员后台当前优先使用聚合接口 `GET /api/v1/admin/console`
- 业务工作区当前优先使用聚合接口 `GET /api/v1/workspace/console`
- 业务工作区接口返回的 `alerts` 只应包含当前登录用户负责的业务告警，不应混入平台事项
- 车辆归属建议以后端真实归属关系为准，前端只负责展示当前用户自己的车辆

## 10. 推荐开发顺序

建议后端按下面顺序开发，前端最容易联调：

1. `POST /api/v1/auth/login`
2. `POST /api/v1/auth/register`
3. `GET /api/v1/auth/me`
4. `GET /api/v1/admin/console`
5. `PATCH /api/v1/admin/users/{userId}`
6. `GET /api/v1/workspace/console`
7. `POST /api/v1/workspace/vehicles`
8. `PUT /api/v1/workspace/vehicles/{vehicleId}`
9. `DELETE /api/v1/workspace/vehicles/{vehicleId}`
10. `PATCH /api/v1/workspace/alerts/{alertId}`

如果后端后续想拆成更细的列表接口，也可以再补充，但以上这些已经足够支撑当前前端页面完整联调。
