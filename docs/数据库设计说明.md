# 数据库设计说明

适用范围：

- 后端
- 前端查询理解
- 算法数据需求确认

关联文件：

- 表结构：`backend-service/sql/schema.sql`
- 种子数据：`backend-service/sql/seed.sql`
- 接口文档：`backend-service/API.md`

## 1. 设计目标

当前数据库设计服务于 3 类需求：

- 后端业务接口查询
- 前端大屏与历史页面展示
- 算法输入输出结果存储

当前项目仍以 mock 数据驱动为主，但 SQL 结构已经按后续 MySQL 落地来设计。

## 2. 设计原则

- 主场景只保留疫苗冷链配送
- 优先满足大屏查询和历史查询
- 风险评估和路径建议单独成表，便于后续算法接入
- 字段命名与接口字段保持强关联，减少映射成本

## 3. 核心表清单

当前核心表共 6 张：

- `vehicles`
- `cargo_profiles`
- `telemetry_records`
- `alerts`
- `risk_assessments`
- `route_plans`

## 4. 表用途说明

### 4.1 `vehicles`

用途：

- 存储车辆基础信息
- 作为大屏首页、车辆下拉、车辆卡片基础来源

关键字段：

- `vehicle_code`
- `plate_number`
- `cargo_type`
- `cargo_name`
- `safe_temp_min`
- `safe_temp_max`
- `status`

说明：

- `vehicle_code` 是当前项目中的业务主键
- 前端和算法统一通过 `vehicleCode` 识别车辆

### 4.2 `cargo_profiles`

用途：

- 存储货物类型和安全温区配置
- 后续可支持多货物场景

当前状态：

- MVP 阶段只保留 `VACCINE`

关键字段：

- `cargo_type`
- `cargo_name`
- `safe_temp_min`
- `safe_temp_max`
- `sensitivity_level`

### 4.3 `telemetry_records`

用途：

- 存储车辆温湿度、车门状态、速度、位置等时序数据
- 支撑最新状态接口和历史趋势接口

关键字段：

- `vehicle_code`
- `record_time`
- `temperature`
- `humidity`
- `door_open`
- `speed`
- `outside_temp`
- `lng`
- `lat`
- `remaining_km`
- `trend`

前端依赖：

- 温度折线图
- 车辆状态卡片
- 地图当前位置

算法依赖：

- 连续温度序列
- 车门开启影响
- 剩余路程与风险相关性

### 4.4 `alerts`

用途：

- 存储预警与告警事件
- 支撑告警卡片、历史告警页、告警详情

关键字段：

- `alert_id`
- `vehicle_code`
- `alert_level`
- `alert_type`
- `title`
- `detail_text`
- `suggestion`
- `trigger_time`
- `status`

说明：

- `alert_id` 是告警业务唯一编号
- `status` 为后续“已确认 / 已关闭”预留

### 4.5 `risk_assessments`

用途：

- 存储风险评估结果
- 支撑风险卡片、风险历史查询、算法结果落库

关键字段：

- `vehicle_code`
- `risk_score`
- `risk_level`
- `risk_label`
- `risk_reason`
- `predicted_minutes_to_limit`
- `algorithm_version`
- `algorithm_source`
- `assessment_time`

说明：

- 这是第 2 周重点补齐的分析结果表
- 后续真实算法接入时，建议每次评估都落一条记录

### 4.6 `route_plans`

用途：

- 存储调度建议或重规划结果
- 支撑前端展示“当前推荐方案”和“候选方案列表”

关键字段：

- `vehicle_code`
- `plan_type`
- `plan_title`
- `plan_detail`
- `estimated_cost`
- `estimated_benefit`
- `recommended`
- `created_time`

说明：

- 一辆车可对应多条候选方案
- `recommended=1` 表示推荐方案

## 5. 表关系理解

当前可按下面关系理解：

- `vehicles` 1 对多 `telemetry_records`
- `vehicles` 1 对多 `alerts`
- `vehicles` 1 对多 `risk_assessments`
- `vehicles` 1 对多 `route_plans`

当前 SQL 中没有强制外键，原因是：

- MVP 阶段优先保证开发灵活
- 便于先用 mock 数据和脚本快速落数据

如果后续接真实 MySQL，可再补正式外键约束。

## 6. 当前数据库如何支撑接口

### 6.1 大屏聚合接口

- 接口：`/api/v1/dashboard/vehicles/{vehicleCode}`

主要依赖：

- `vehicles`
- `telemetry_records`
- `alerts`
- `risk_assessments`
- `route_plans`

### 6.2 历史温度接口

- 接口：`/api/v1/vehicles/{vehicleCode}/telemetry/history`

主要依赖：

- `telemetry_records`

### 6.3 告警接口

- 接口：
  - `/api/v1/vehicles/{vehicleCode}/alerts`
  - `/api/v1/alerts`
  - `/api/v1/alerts/{alertId}`

主要依赖：

- `alerts`

### 6.4 风险评估接口

- 接口：
  - `/api/v1/vehicles/{vehicleCode}/risk-assessments/latest`
  - `/api/v1/vehicles/{vehicleCode}/risk-assessments/history`

主要依赖：

- `risk_assessments`

### 6.5 路径建议接口

- 接口：
  - `/api/v1/vehicles/{vehicleCode}/route-plans/latest`
  - `/api/v1/vehicles/{vehicleCode}/route-plans`

主要依赖：

- `route_plans`

## 7. 当前数据库对前端的价值

前端不需要直接操作数据库，但需要理解这些事实：

- 大屏不是只查一张表
- 风险与路径建议来自独立分析结果表
- 历史页和卡片页的数据来源不同

这会帮助前端理解为什么后端会提供：

- 单接口查询
- 聚合接口查询
- 历史接口查询

## 8. 当前数据库对算法的价值

算法同学需要重点关注：

- `telemetry_records`：输入序列
- `alerts`：异常场景标签参考
- `risk_assessments`：输出结果落库位置
- `route_plans`：调度建议落库位置

也就是说：

- 训练或推理输入主要来自 `telemetry_records`
- 推理输出主要落到 `risk_assessments` 和 `route_plans`

## 9. 后续建议

数据库后续建议按这个顺序推进：

1. 用 MySQL 正式落表
2. 用真实 repository 替换 mock repository
3. 增加 `destinations` 表
4. 增加 `algorithm_call_logs` 表
5. 增加告警确认人与处理记录表

## 10. 当前结论

这套数据库设计已经足够支撑：

- 第 1 周后端基础接口
- 第 2 周风险评估与路径建议接口
- 前后端联调
- 算法接入边界明确化

如果后续规模不扩大，这套设计已经能完整支撑课程项目演示。
