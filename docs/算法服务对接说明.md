## 1. 架构总览

```
┌──────────────┐     HTTP/JSON      ┌──────────────────┐
│  Python 算法  │ ◄───────────────► │   Java 后端       │
│  服务 :5001   │    evaluate       │   :18081          │
│              │    /simulation/    │                   │
│              │    telemetry       │                   │
└──────────────┘                    │  ┌─────────────┐  │
                                    │  │ MockEngine  │  │
                                    │  │ (降级/开发)  │  │
                                    │  └─────────────┘  │
                                    │        │          │
                                    │  ┌─────▼────────┐ │
                                    │  │  MySQL       │ │
                                    │  │ risk_assess..│ │
                                    │  │ route_plans  │ │
                                    │  │ telemetry_.. │ │
                                    │  └──────────────┘ │
                                    └──────────────────┘
```

核心设计原则：

- 算法服务是独立进程，后端通过 HTTP/JSON 调用
- 算法不直接写业务数据库，只负责计算和返回结果
- 后端负责编排输入、持久化输出、接口暴露
- mock 引擎作为开发期和降级期的兜底方案

## 2. 网关模式

### 2.1 接口定义

`AlgorithmGateway` 是算法调用的统一入口：

```java
public interface AlgorithmGateway {
    AlgorithmEvaluation evaluate(vehicleCode, vehicle, latestTelemetry,
        telemetryHistory, alerts);
    AlgorithmGatewayStatus status();
}
```

所有调用者只依赖接口，不感知底层是 mock 还是 HTTP。

### 2.2 可切换网关

`SwitchableAlgorithmGateway` 根据配置决定走哪条链路：

| 配置值 | 行为 |
|--------|------|
| `mock` | 直接走 MockAlgorithmEngine，不发起 HTTP |
| `http` | 走 HttpAlgorithmGateway，失败时如果 `fallback-enabled=true` 则降级到 MockAlgorithmEngine |
| `http-gateway`（默认） | 与 `http` 相同 |

降级策略：HTTP 调用抛出 `AlgorithmGatewayException` 时，若 `fallback-enabled=true`，自动切换到 MockAlgorithmEngine 返回兜底结果，保证前端不崩溃。

### 2.3 Mock 算法引擎

`MockAlgorithmEngine` 提供基于规则的评估：

- 温度 ≥ 6.5 或已有告警 ≥ 2 条 → 判定异常
- 风险分数 = 温度偏差分 + 剩余路程分 + 车门开启分 + 告警数量分
- 分数 ≥ 80 → `HIGH`，≥ 45 → `MEDIUM`，否则 `LOW`
- 高风险 → 建议改道冷库或优先配送
- 中风险 → 建议优先配送或检查制冷设备
- 低风险 → 建议按原计划配送

用于开发期快速验证和 HTTP 故障时的兜底。

### 2.4 HTTP 算法网关

`HttpAlgorithmGateway` 负责与 Python 算法服务通信：

- **evaluate 请求**：`POST {base-url}/evaluate`，body 为 `AlgorithmEvaluateRequest` JSON
- **health 检查**：`GET {base-url}/health`，用于 `status()` 判断服务可达性
- 超时或非 2xx 响应 → 抛出 `AlgorithmGatewayException`

### 3.3 枚举约定

风险等级（`riskLevel`）只允许：`LOW`、`MEDIUM`、`HIGH`

时间格式统一为：`yyyy-MM-dd HH:mm:ss`

布尔值统一为 JSON `true`/`false`，禁止 `0`/`1`/`"yes"`/`"no"`

## 4. 配置参考

`application.yml` 中的算法相关配置：

```yaml
app:
  algorithm:
    mode: http-gateway       # mock | http | http-gateway
    version: mock-risk-v1    # 算法版本标识（mock 默认值）
    fallback-enabled: true   # HTTP 失败时是否降级到 mock
    base-url: http://localhost:5001          # Python 算法服务地址
    evaluate-path: /evaluate                 # 评估接口路径
    simulation-telemetry-path: /simulation/telemetry  # 模拟遥测接口路径
```

- **mock 模式开发**：`mode: mock`，完全不依赖外部算法服务
- **联调真实算法**：`mode: http`，设置 `base-url` 指向算法服务地址
- **生产建议**：`mode: http` + `fallback-enabled: true`，保证算法故障时服务降级可用

## 5. 数据同步流程（MySQL 模式）

以下流程仅在 `@Profile("mysql")` 下生效，通过 `SimulationController` 手动触发。

### 5.1 模拟遥测同步

`POST /api/v1/simulation/import-telemetry`

```
后端 ──GET──► 算法服务 /simulation/telemetry
       ◄── JSON { vehicleCode: [telemetry...] }
              │
              ▼
       解析 → TelemetryRecordEntity → saveAll() → telemetry_records 表
```

### 5.2 算法分析与落库

`POST /api/v1/simulation/import-analysis`

```
后端 ──for each vehicle──► algorithmGateway.evaluate()
              │
              ▼
       AlgorithmEvaluation
         ├── riskScore/riskLevel/riskReason → RiskAssessmentEntity → risk_assessments 表
         └── recommendations[] → RoutePlanEntity → route_plans 表
```

每次调用会先清空 `risk_assessments` 和 `route_plans` 表再全量写入。

### 5.3 触发方式

两个接口均需手动 POST 调用，未接入定时任务。建议联调顺序：

1. 确保算法服务启动（`GET /health` 可达）
2. `POST /api/v1/simulation/import-telemetry` — 拉取遥测数据入库
3. `POST /api/v1/simulation/import-analysis` — 逐车调用 evaluate 并落库
4. 前端刷新大屏即可看到风险评估与路径建议结果

## 6. 关键 Java 类型

| 类 | 职责 |
|----|------|
| `AlgorithmGateway` | 算法调用统一接口 |
| `SwitchableAlgorithmGateway` | 根据配置切换 mock/HTTP，实现降级 |
| `MockAlgorithmEngine` | 纯 Java 规则引擎，开发期和降级兜底 |
| `HttpAlgorithmGateway` | HTTP 调用真实 Python 算法服务 |
| `AlgorithmEvaluation` | 算法评估结果 record（riskScore, riskLevel, recommendations...） |
| `AlgorithmRecommendation` | 单条路径建议 record（planType, title, detail...） |
| `AlgorithmGatewayStatus` | 网关状态 record（mode, available, version...） |
| `AlgorithmGatewayException` | 算法调用异常 |
| `AlgorithmSimulationSyncService` | 从算法服务拉遥测数据写入 `telemetry_records` |
| `AlgorithmAnalysisSyncService` | 调用 evaluate 并将结果写入 `risk_assessments` + `route_plans` |
| `SimulationController` | 暴露 `/api/v1/simulation/*` 同步入口 |

## 7. 添加新算法能力的步骤

当需要让算法输出新字段（例如新异常类型、新建议类型）：

1. 在 `AlgorithmEvaluation` / `AlgorithmRecommendation` record 中加字段
2. 在 `HttpAlgorithmGateway.AlgorithmHttpResponse` 中加对应字段
3. 在 `MockAlgorithmEngine` 中提供 mock 默认值
4. 在 `AlgorithmAnalysisSyncService.syncAnalysisToDatabase()` 中处理新字段的落库
5. 如需新表，在 `sql/schema.sql` 中建表，添加对应 JPA Entity 和 Repository
6. 更新 `docs/算法输入输出规范.md` 字段说明

## 8. 常见问题

**Q: mock 模式下算法相关的接口能调吗？**

A: 能。mock 模式走 `MockAlgorithmEngine`，不依赖外部服务。但 `SimulationController`（`/api/v1/simulation/*`）只在 `@Profile("mysql")` 下注册，mock 模式下不可用。

**Q: 如何验证当前走的是 mock 还是真实算法？**

A: `GET /api/v1/algorithm/status`，返回的 `mode` 和 `algorithmSource` 字段会标明来源。

**Q: 算法服务返回 500 怎么办？**

A: 若 `fallback-enabled=true`，自动降级到 `MockAlgorithmEngine`，前端正常展示兜底数据。检查日志中的 `AlgorithmGatewayException` 定位根因。

**Q: 算法同学需要了解 Java 代码吗？**

A: 不需要。算法同学只需关注 HTTP 契约（第 3 节），实现 `POST /evaluate` 和 `GET /simulation/telemetry` 两个端点即可。
