# 接口联调说明

适用范围：

- 前端与后端联调
- 后端与算法联调
- 当前项目 MVP 阶段

关联文档：

- 后端接口文档：`backend-service/API.md`
- 项目 PRD：`docs/PRD-冷链运输温控预警平台-MVP.md`

## 1. 联调目标

当前阶段联调目标是：

- 前端不再依赖硬编码假数据
- 前端可以通过后端接口获取车辆、温度、告警和大屏聚合数据
- 后端输出结构稳定，便于算法后续接入
- 算法接入前，后端先用 mock 数据保证页面可演示

## 2. 当前联调范围

本阶段只覆盖以下功能：

- 车辆基础信息查询
- 车辆最新温度数据查询
- 车辆历史温度数据查询
- 告警列表查询
- 告警详情查询
- 大屏聚合接口查询
- 告警汇总查询
- 风险评估结果查询
- 路径建议查询
- 算法网关状态查询

本阶段不覆盖：

- 登录鉴权
- 真正的 MySQL 持久化
- 真实硬件接入
- WebSocket 实时推送
- 多车联合调度

## 3. 服务地址

后端服务默认地址：

- `http://localhost:18081`

关键访问地址：

- 首页：`/`
- 健康检查：`/health`
- 接口文档页：`/api-docs`

接口统一前缀：

- `/api/v1`

## 4. 联调接口清单

### 4.1 获取车辆列表

- Method：`GET`
- Path：`/api/v1/vehicles`
- 用途：页面初始化时获取车辆下拉、车辆卡片基础信息

### 4.2 获取大屏聚合数据

- Method：`GET`
- Path：`/api/v1/dashboard/vehicles/{vehicleCode}`
- 用途：前端大屏优先使用此接口，一次拿到 summary、route、temperatureHistory、alerts

建议前端优先对接此接口，因为它最接近当前大屏原型。

### 4.3 获取车辆最新温度数据

- Method：`GET`
- Path：`/api/v1/vehicles/{vehicleCode}/telemetry/latest`
- 用途：单独刷新某辆车最新状态

### 4.4 获取历史温度数据

- Method：`GET`
- Path：`/api/v1/vehicles/{vehicleCode}/telemetry/history`
- Query：`minutes`
- 用途：温度折线图展示

当前说明：

- 当前 mock 数据固定返回当前车辆的一组历史温度序列
- `minutes` 目前作为兼容字段保留，后续可接真实查询逻辑

### 4.5 获取车辆告警记录

- Method：`GET`
- Path：`/api/v1/vehicles/{vehicleCode}/alerts`
- Query：`limit`
- 用途：告警卡片面板展示

### 4.6 获取分页告警列表

- Method：`GET`
- Path：`/api/v1/alerts`
- Query：
  - `vehicleCode`
  - `page`
  - `pageSize`
- 用途：历史告警页面或列表页

### 4.7 获取告警详情

- Method：`GET`
- Path：`/api/v1/alerts/{alertId}`
- 用途：历史告警详情页、详情弹窗

### 4.8 获取告警汇总

- Method：`GET`
- Path：`/api/v1/vehicles/{vehicleCode}/alerts/summary`
- 用途：快速展示当前车辆告警数量、最高等级、最近一条告警标题

### 4.9 获取最新风险评估结果

- Method：`GET`
- Path：`/api/v1/vehicles/{vehicleCode}/risk-assessments/latest`
- 用途：风险卡片、风险状态展示

### 4.10 获取风险评估历史

- Method：`GET`
- Path：`/api/v1/vehicles/{vehicleCode}/risk-assessments/history`
- Query：`limit`
- 用途：风险历史列表、风险趋势分析

### 4.11 获取最新路径建议

- Method：`GET`
- Path：`/api/v1/vehicles/{vehicleCode}/route-plans/latest`
- 用途：当前调度建议展示

### 4.12 获取路径建议列表

- Method：`GET`
- Path：`/api/v1/vehicles/{vehicleCode}/route-plans`
- Query：`limit`
- 用途：查看当前车辆的多条候选方案

### 4.13 获取算法网关状态

- Method：`GET`
- Path：`/api/v1/algorithm/status`
- 用途：联调时确认当前是否为 mock 算法网关、是否启用降级

## 5. 前端联调约定

### 5.1 前端优先使用的接口

首页大屏建议优先使用：

- `GET /api/v1/dashboard/vehicles/{vehicleCode}`

如果后续做局部刷新，再补充：

- `GET /api/v1/vehicles/{vehicleCode}/telemetry/latest`
- `GET /api/v1/vehicles/{vehicleCode}/alerts?limit=4`
- `GET /api/v1/vehicles/{vehicleCode}/alerts/summary`
- `GET /api/v1/vehicles/{vehicleCode}/risk-assessments/latest`
- `GET /api/v1/vehicles/{vehicleCode}/route-plans/latest`

### 5.2 轮询建议

当前建议：

- 大屏轮询频率：`5 秒`

原因：

- 当前是课程项目 MVP
- 轮询比 WebSocket 更简单稳定
- 足够支撑演示效果

### 5.3 当前推荐的默认车辆

- `CC-VA-01`

说明：

- 当前这辆车是主演示车
- 温度历史和告警场景最完整
- 前端大屏联调请优先使用它

## 6. 算法联调约定

当前后端尚未接入真实算法服务，但先约定边界：

- 算法服务协议：HTTP/JSON
- 当前后端先通过 mock 算法网关输出风险与路线建议结构
- 算法接入时，优先保持字段名不变

后续算法接入建议：

- 输入由后端组织
- 输出由后端持久化和二次封装
- 前端不直接调用算法服务
- 当前可先通过 `/api/v1/algorithm/status` 判断后端是否处于 mock 模式

## 7. 通用返回格式

所有接口统一返回：

```json
{
  "success": true,
  "message": "ok",
  "data": {}
}
```

返回约定：

- `success=true` 表示请求成功
- `message=ok` 表示本次请求正常
- `data` 为具体业务对象、数组或详情结构

## 8. 联调注意事项

### 8.1 不允许随意修改公共字段名

例如：

- 不要把 `vehicleCode` 改成 `vehicleId`
- 不要把 `triggerTime` 改成 `alertTime`
- 不要把 `doorOpen` 改成 `doorStatus`

如果必须修改：

- 先通知前端和算法
- 再同步修改接口文档和字段规范文档

### 8.2 不允许前端自行发明返回结构

前端展示层可以做转换，但不要反向要求后端临时改字段以适配个别组件。

### 8.3 不允许算法直接写业务数据库

算法只负责：

- 输入处理
- 模型输出
- 风险结果
- 建议结果

后端负责：

- 编排
- 存储
- 接口输出

## 9. 联调问题排查顺序

当前如果联调失败，建议按这个顺序排查：

1. 后端服务是否启动
2. `GET /health` 是否返回 `UP`
3. 路径是否带了 `/api/v1`
4. `vehicleCode` 是否写成了 `CC-VA-01`
5. 前端请求端口是否是 `18081`
6. 请求方法是否正确
7. 接口字段是否按照字段字典解析

## 10. 变更同步规则

涉及以下改动时，后端必须同步更新文档：

- 新增接口
- 删除接口
- 修改字段名
- 修改枚举值
- 修改时间格式
- 修改返回结构

必须同步更新的文档：

- `backend-service/API.md`
- `docs/字段与枚举规范.md`
- 如影响范围较大，同时更新 `docs/接口联调说明.md`
