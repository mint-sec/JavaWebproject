# 团队Git协作规范文档

# 1\. 协作说明

本文档为项目通用Git提交规范，适用于本仓库所有开发人员。规范目的为统一代码提交流程，避免代码覆盖、代码冲突、版本错乱问题，保障团队开发稳定进行。

# 2\. 开发分工规则

1. **文件权责划分**：开发人员仅允许修改个人负责的业务文件，禁止擅自修改他人编辑文件。

2. **配置文件权限**：项目配置类文件（pom\.xml、web\.xml、yml、properties等）仅由仓库管理员维护，普通成员禁止改动。

3. **修改限制**：多人尽量避免修改同一文件，从根源减少代码冲突。

# 3\. 标准提交流程

所有人员提交代码必须严格遵循以下固定顺序，不可简化步骤。

## 3\.1 开发前操作

拉取远程最新代码，同步本地项目版本：

```bash
git pull origin main
```

## 3\.2 开发后操作

1\. 暂存本地修改文件

```bash
git add .
```

2\. 提交本地修改，备注需清晰标注修改内容

```bash
git commit -m "填写本次修改内容"
```

3\. 推送前再次拉取远程代码，合并他人更新内容

```bash
git pull origin main
```

4\. 推送本地代码至远程仓库

```bash
git push origin main
```

# 4\. 原理说明

若两名开发人员初始拉取同一版本代码，A优先推送更新，B本地保留旧版本。B若直接推送，旧版本文件会覆盖远程新版代码。推送前执行 `git pull`，可自动合并远程最新修改，保留所有人代码，规避覆盖风险。

**补充：修改不同文件时，Git自动合并代码，无冲突风险；修改同一文件极易产生冲突。**

# 5\. 禁止操作条例

1. 禁止使用强制推送命令 `git push \-f`。

2. 禁止随意执行版本回退命令 `git reset`。

3. 禁止直接拖拽替换项目文件夹。

4. 遇到报错、冲突，禁止私自修改代码处理，需告知管理员统一解决。

# 6\. 常见问题

### 6\.1 不同文件修改是否会互相覆盖？

不会，Git 会自动合并不同文件的修改，保留全部代码。

### 6\.2 本地为旧版本代码能否直接推送？

不可以，必须先拉取远程最新代码，合并后方可推送。

### 6\.3 出现代码冲突如何处理？

保留冲突界面，截图反馈给仓库管理员，由管理员统一合并处理。
